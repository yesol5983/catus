# Catus 앱 출시 전략 및 플레이스토어 론칭 계획

이 문서는 현재 React로 개발된 Catus 웹 애플리케이션을 모바일 앱으로 확장하고 구글 플레이스토어에 출시하기 위한 전략을 분석하고, 구체적인 실행 계획을 정리한 것입니다.

## 1. 앱 출시 전략 비교 분석

기존 React 프로젝트를 모바일 앱으로 만드는 세 가지 주요 방법(PWA, Capacitor, React Native)을 비교하고 Catus에 가장 적합한 방식을 도출합니다.

| 구분 | PWA (Progressive Web App) | Capacitor (캐패시터) | React Native (리액트 네이티브) |
| :--- | :--- | :--- | :--- |
| **개념** | 웹 기술로 앱처럼 보이게 만드는 웹 표준. TWA로 스토어 등록. | 기존 웹 앱을 네이티브 컨테이너로 감싸고, 다양한 네이티브 기능을 연결. | 네이티브 UI 컴포넌트로 직접 렌더링하는 앱 개발 프레임워크. |
| **장점** | **가장 쉬움**. 코드 변경 거의 없음. 웹/앱 100% 동일 코드. | **쉬운 적용**과 **높은 확장성**의 균형. 코드 재사용 극대화. 풍부한 네이티브 기능. | **최고의 성능/UX**. 완벽한 네이티브 기능 활용. |
| **단점** | 네이티브 경험/성능 한계. 기능 제약 많음. | 웹뷰 기반의 성능 한계 존재. | **사실상 재개발**. UI 코드를 전부 새로 작성해야 함. |
| **추천 대상** | 가장 빠르게 스토어에 등록하는 것이 유일한 목표일 때. | 기존 웹 프로젝트를 활용하여, 점진적으로 네이티브 기능을 추가하고 싶을 때. | 시간과 비용을 투자하여 최고의 네이티브 앱을 만들고 싶을 때. |

### **결론: Catus를 위한 최적의 선택**

Catus 프로젝트의 경우, **Capacitor(캐패시터)를 사용하는 것이 가장 이상적입니다.**

*   **이유**: 이미 완성된 웹 코드를 대부분 재사용하여 **쉽게 시작**할 수 있고, PWA의 한계를 넘어 카메라, 파일 시스템, 생체 인증 등 **다양한 네이티브 기능을 손쉽게 확장**할 수 있습니다. Catus와 같은 콘텐츠 및 채팅 중심 앱에서는 Capacitor가 제공하는 성능으로도 충분히 훌륭한 사용자 경험을 제공할 수 있습니다.

## 2. Capacitor 기반 구글 플레이스토어 출시 계획

Capacitor 적용부터 스토어 출시에 이르는 상세한 단계별 계획입니다.

### **(A) Capacitor 프로젝트 설정**

1.  **Capacitor 설치**: 프로젝트에 관련 패키지를 설치합니다.
    ```bash
    npm install @capacitor/core
    npm install @capacitor/cli --save-dev
    ```
2.  **Capacitor 초기화**: `npx cap init` 명령어로 프로젝트 설정을 시작하고 앱 이름(예: `Catus`)과 ID(예: `com.catus.app`)를 지정합니다.
3.  **웹 빌드 경로 설정**: 생성된 `capacitor.config.ts` 파일에서 `webDir` 속성을 Vite의 빌드 결과물 폴더인 `'dist'`로 수정합니다.
4.  **안드로이드 플랫폼 추가**: `npx cap add android` 명령어로 `android` 네이티브 프로젝트 폴더를 생성합니다.

### **(B) 앱 완성도 높이기**

1.  **앱 아이콘 및 스플래시 스크린**: `trapeze`나 `cordova-res`와 같은 도구를 사용하여 모든 기기 사이즈에 맞는 앱 아이콘과 앱 실행 시 로딩 화면을 생성 및 적용합니다. 이는 앱의 첫인상을 결정하는 중요한 요소입니다.
2.  **네이티브 UI 조정**: `@capacitor/status-bar` 플러그인 등을 사용하여 앱 상단 상태바의 색상이나 스타일을 앱의 전반적인 디자인과 통일시킵니다.

### **(C) 출시용 파일 준비 및 스토어 등록**

1.  **앱 서명 키 생성**: 안드로이드 스튜디오(`Build > Generate Signed Bundle / APK...`)를 통해 앱의 소유권을 증명하는 서명 키(.jks)를 생성합니다. **이 키는 분실 시 앱 업데이트가 절대 불가능하므로, 반드시 안전하게 백업해야 합니다.**
2.  **출시용 앱 빌드(AAB)**: 위에서 생성한 서명 키를 사용하여, 구글 플레이스토어가 요구하는 **AAB(Android App Bundle)** 형식으로 앱을 빌드합니다.
3.  **구글 개발자 계정 등록**: 구글 플레이 콘솔에서 개발자 계정을 생성합니다 (최초 1회, $25 비용 발생).
4.  **스토어 리스팅 정보 작성**:
    *   **텍스트**: 앱 이름, 간단한 설명, 자세한 설명
    *   **그래픽**: 고해상도 아이콘(512x512), 피처 그래픽(1024x500), 스마트폰/태블릿용 스크린샷 등
5.  **개인정보처리방침 준비**: Catus는 사용자 데이터를 다루므로, **개인정보처리방침을 설명하는 웹페이지를 준비하고 URL을 제출**해야 합니다.
6.  **앱 콘텐츠 정보 입력**: 앱의 대상 연령, 카테고리 등 정책 관련 설문에 답변합니다.

### **(D) 테스트 및 출시**

1.  **테스트 트랙 활용**:
    *   **참고사항**: 기존에 웹에서 테스트를 진행한 30명의 테스터 풀이 있으므로, 이들을 적극 활용하는 것이 좋습니다.
    *   **실행방안**: '프로덕션'으로 바로 출시하기 전에, '내부 테스트' 또는 '비공개 테스트' 트랙에 테스터들을 초대하여 스토어 버전 앱의 안정성을 최종 검증합니다.
2.  **프로덕션 출시**: 테스트가 완료되면 프로덕션 트랙으로 전환하여 전체 사용자에게 앱을 공개합니다.
3.  **구글 앱 검토**: 출시 요청 후 구글의 검토를 기다립니다. 신규 앱의 최초 검토는 보통 며칠 이상 소요될 수 있습니다.

이 계획을 따르면, 기존 Catus 프로젝트를 효율적으로 활용하여 완성도 높은 앱을 구글 플레이스토어에 성공적으로 출시할 수 있습니다.
